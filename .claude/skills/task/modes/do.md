# Режим `do` — выполнение задачи

Цель: взять задачу со статусом `ready`, выполнить её план, обновить `TASK.md`, довести до статуса `review`.

**Git не трогать. Коммиты делает пользователь вручную.**

## Шаги

### 1. Определение задачи

- Если после `do` указан ID (`T-NNNN`) — используй его.
- Если ID не указан:
  1. Прочитай `.planning/tasks/INDEX.md`.
  2. Отфильтруй задачи со статусом `ready`.
  3. Если таких нет — сообщи: «Нет задач со статусом `ready`. Создай через `/task new` или укажи ID явно.» Остановись.
  4. Если есть — покажи списком с номерами:
     ```
     Задачи в статусе ready:
     1. T-0001 «Движение корабля»
     2. T-0003 «Стрельба»
     
     Какую выполняем? (номер или ID)
     ```
     Жди ответа.

- Проверь, что файл `.planning/tasks/T-NNNN-<slug>/TASK.md` существует. Если нет — проверь archive/. Если и там нет — сообщи пользователю и остановись.

### 2. Загрузка контекста

1. Прочитай `.planning/tasks/T-NNNN-<slug>/TASK.md` целиком.
2. Если в секции «Связанные задачи» указаны другие T-NNNN — прочитай их `TASK.md` (и в archive/ тоже посмотри).
3. Если в секции «Изученная документация» указаны файлы — убедись, что ознакомлен с релевантными. Если файл критичен для текущих шагов плана — прочитай его заново.

### 3. Смена статуса на executing

1. В frontmatter `TASK.md` замени `status: ready` → `status: executing`.
2. В секции `## История статусов` добавь строку:
   ```
   - 2026-04-18 HH:MM executing
   ```
   (текущая дата из контекста + примерное время, если оно уместно; если часы неизвестны — просто дата).
3. В `.planning/tasks/INDEX.md` в строке задачи замени статус `ready` → `executing`.

### 4. Выполнение пунктов плана

Для КАЖДОГО пункта `- [ ]` в секции `## План`:

1. **Выполни действия пункта** — используй Edit/Write/Bash по необходимости.
2. **Помечай выполненное** — в `TASK.md` замени `- [ ]` → `- [x]` для этого пункта.
3. **Запись в журнал** — в секцию `## Журнал выполнения` добавь одну-две строки:
   ```
   - 2026-04-18 — Шаг N: краткое описание что сделано. Файлы: path/to/file.cs, path/to/other.cs
   ```
4. **Проверки CLAUDE.md** — если меняешь `.cs` код, соблюдай правила из `CLAUDE.md`: компиляция через MCP, тесты EditMode+PlayMode перед завершением задачи. Но это проверяется автоматически, не спрашивая пользователя.

### 5. Обработка вопросов во время выполнения

Если сталкиваешься с неоднозначностью, которая не была обсуждена в режиме `new`:

**НЕ УГАДЫВАЙ. НЕ ПРИНИМАЙ РЕШЕНИЕ САМ.**

1. Останови выполнение (не делай edit'ы в коде до ответа).
2. Добавь в секцию `## Обсуждение` TASK.md запись:
   ```
   - Q (2026-04-18): <чёткая формулировка вопроса с контекстом — что непонятно и какие варианты>
     A: <пусто>
   ```
3. В этой же правке TASK.md — в `## Журнал выполнения` добавь строку:
   ```
   - 2026-04-18 — Пауза: открыт вопрос Q-N (см. Обсуждение).
   ```
4. Задай вопрос пользователю в чате.
5. Жди ответ.
6. Когда пришёл — заполни `A:` в TASK.md.
7. При необходимости обнови секцию `## Решения` и/или `## План` (если план меняется — обнови и INDEX.md не требуется, структура задачи не меняется).
8. Продолжи выполнение.

### 6. Завершение

Когда все пункты плана помечены `[x]`:

1. В frontmatter: `status: executing` → `status: review`.
2. В `## История статусов`: добавь `- 2026-04-18 HH:MM review`.
3. В INDEX.md: замени статус `executing` → `review`.
4. Сообщи пользователю:
   ```
   Задача T-NNNN выполнена. Статус: review.
   Изменённые файлы (сводка):
   - path/to/file1.cs
   - path/to/file2.cs
   
   Проверь результат и запусти /task review T-NNNN.
   ```

## Запрещено в этом режиме

- Git-операции любого рода.
- Менять `## План` молча — если план корректируется из-за вопроса, это фиксируется в `## Решения`.
- Продолжать выполнение при незакрытом вопросе в `## Обсуждение`.
- Переводить задачу в `done` — это делает только режим `review` после явного одобрения пользователя.
- Создавать новые задачи — если обнаружено, что часть работы относится к отдельной задаче, запиши это в `## Журнал выполнения` и спроси пользователя: «Выделить это в отдельную задачу через `/task new`?».
