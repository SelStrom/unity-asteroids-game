# Режим `review` — проверка и доработки

Цель: показать пользователю результат выполнения → принять правки или одобрение → при правках применить их и вернуться к проверке → при одобрении перевести в `done`.

**Git не трогать. Пользователь сам коммитит результат, когда готов.**

## Шаги

### 1. Определение задачи

- Если после `review` указан ID — используй его.
- Если ID не указан:
  1. Прочитай `.planning/tasks/INDEX.md`.
  2. Отфильтруй задачи со статусом `review` или `revise`.
  3. Если таких нет — сообщи: «Нет задач для проверки. Доступные `ready` задачи запускай через `/task do`.» Остановись.
  4. Покажи списком с номерами, жди выбора.

- Прочитай `TASK.md` выбранной задачи.

### 2. Резюме для пользователя

Покажи одним сообщением:

```
T-NNNN «<заголовок>» — статус <review|revise>

Что сделано:
- <Шаг 1 из плана>
- <Шаг 2 из плана>
- ...

Изменённые файлы (из журнала):
- path/to/file1.cs
- path/to/file2.cs

Решения по ходу выполнения (если были):
- <краткая выжимка из Обсуждения/Решений, если добавлялись во время do>

Всё ок — напиши «принято».
Нужны доработки — перечисли их.
```

Жди ответа.

### 3. Развилка

Посмотри на ответ пользователя:

- Слова **«принято», «ок», «готово», «done», «approve»** (в любой форме), без списка правок → ветка 3а.
- Перечислены правки (список, текст с требованиями изменений) → ветка 3б.
- Смешанный ответ (часть принято, часть правок) → считай как 3б, применяй правки.

### 3а. Ветка: приёмка

1. В frontmatter TASK.md: `status: review` → `status: done`.
2. В `## История статусов`: добавь `- 2026-04-18 HH:MM done`.
3. В INDEX.md: замени статус `review` → `done`.
4. **Предложи синхронизировать GSD-документы:** спроси «Запустить `/task updocs T-NNNN` сейчас, чтобы обновить PROJECT.md / REQUIREMENTS.md / ROADMAP.md / STATE.md? (да/нет)». Жди ответа.
   - **да** → `Read` `.claude/skills/task/modes/updocs.md` и запусти режим `updocs` с данным `T-NNNN`. После завершения `updocs` вернись к шагу 5.
   - **нет** (или «позже») → продолжай к шагу 5.
5. Спроси пользователя: «Архивировать задачу T-NNNN? (перенести папку в `.planning/tasks/archive/`)». Жди ответа.
6. Если да:
   - `Bash`: `mv .planning/tasks/T-NNNN-<slug> .planning/tasks/archive/` (создай archive/, если её нет: `mkdir -p .planning/tasks/archive`).
   - В INDEX.md: удали строку задачи из основной таблицы. Если в шаблоне INDEX есть секция «Архивные» — добавь туда строку; если нет — просто удали из активных.
7. Если нет — оставь в `.planning/tasks/`.
8. Сообщи: «T-NNNN закрыта (done).» (+ «GSD-документы синхронизированы.» если запускался updocs, + «Перенесена в архив.» если переносили).

### 3б. Ветка: доработки

1. В frontmatter TASK.md: `status: review` → `status: revise`. (Если был уже `revise` — оставь.)
2. В INDEX.md: обнови статус на `revise`.
3. В `## История статусов`: добавь `- 2026-04-18 HH:MM revise`.
4. В секцию `## Доработки` TASK.md добавь для каждой правки запись. Нумерация сквозная: R1, R2, R3… ищи последний номер в файле и продолжай:
   ```
   - R<N> (2026-04-18): <формулировка правки от пользователя, дословно или чуть очищенная>
     Fix: <пусто>
   ```
5. **Перед применением правок — если одна из них не вполне ясна**, задай уточняющий вопрос пользователю. Добавь его в `## Обсуждение` с пометкой `(по R<N>)`. Жди ответ. Заполни.
6. Для каждой правки R<N>:
   - Примени изменения в коде/документации (Edit/Write).
   - Заполни `Fix:` у R<N>:
     ```
     Fix: <краткое описание что сделано>. Файлы: path/to/file.cs, ...
     ```
   - В `## Журнал выполнения` добавь строку:
     ```
     - 2026-04-18 — R<N>: <что сделано>. Файлы: ...
     ```
7. Если меняешь `.cs` код — применяй правила CLAUDE.md (компиляция+тесты через MCP перед завершением).
8. Когда все правки применены:
   - В frontmatter: `status: revise` → `status: review`.
   - В INDEX.md: обнови статус на `review`.
   - В `## История статусов`: добавь `- 2026-04-18 HH:MM review`.
9. Вернись к шагу 2 (показать новое резюме, упомянуть применённые R1..RN, спросить про следующую итерацию).

## Запрещено в этом режиме

- Git-операции (add, commit, push, stash и т.д.).
- Закрывать задачу как `done` без явного одобрения от пользователя на текущей итерации.
- Применять правки молча, без записи в `## Доработки`.
- Менять содержимое уже применённых R<N> записей (они — история). Если та же правка всплыла заново — это новая R<N+1>.
- Удалять файл `TASK.md` или папку задачи при архивации — только `mv`. История не теряется.
