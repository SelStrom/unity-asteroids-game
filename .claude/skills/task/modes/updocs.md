# Режим `updocs` — обновление GSD-документов по результатам задачи

Цель: после закрытия задачи (`done`) синхронизировать общие GSD-документы проекта — перенести статус/требования/журнал активности так, чтобы PROJECT / REQUIREMENTS / ROADMAP / STATE отражали реальное состояние.

**Git не трогать.** **Только общая документация** — frontmatter/структура task-системы не меняется.

---

## Роли документов (справочник для агента)

### `.planning/PROJECT.md` — «что мы строим и зачем»

Высокоуровневое описание продукта. Три списка требований:

- **Validated** — требования, проверенные в бою (демо / playtest / shipped feature).
- **Active** — в работе или запланированные.
- **Out of Scope** — явно вынесенное за рамки текущего milestone.

Плюс core value, контекст, референсы. Обновляется **редко** — только когда:
- Требование меняет статус Active ↔ Validated (подтвердилось / отменилось).
- Появилось новое требование верхнего уровня (не INFRA/SHIP-ID, а формулировка).
- Изменился scope milestone (core value / out of scope).

### `.planning/REQUIREMENTS.md` — детальный контракт

Каждое требование имеет ID формата `<CATEGORY>-NN` (например, `INFRA-01`, `SHIP-02`, `MAP-03`, `CMBT-01`). Группы: Infrastructure, Ship & Movement, Map & Navigation, Combat, Inventory/Equipment, и т.д. Статус чекбоксом `[x]`/`[ ]`.

Обновляется, когда задача **закрыла конкретный requirement-ID** — отмечаем чекбокс. Если требование вышло частично — оставляем `[ ]` и описываем что покрыто в журнале ниже по файлу.

### `.planning/ROADMAP.md` — план выполнения по фазам

Нумерация: целые (1, 2, 3) — плановые фазы milestone'а, десятичные (2.1, 02.3) с меткой `[INSERTED]` — urgent-вставки. Чекбокс `[x]` = фаза завершена.

Обновляется, когда:
- Задача закрыла **целую фазу или значимую её часть** — отметить `[x]` или оставить `[ ]` с пометкой о частичном прогрессе.
- Появилась urgent-работа, которую стоит зафиксировать как decimal-фазу (например, task-цепочка T-0002..T-0005 по сути = INSERTED-фаза).

### `.planning/STATE.md` — живой tracker

YAML frontmatter (`milestone`, `status`, `last_updated`, `last_activity`, `progress.{total_phases, completed_phases, total_plans, completed_plans, percent}`) + markdown-секции (Current Position, Performance Metrics, Last Activity).

Обновляется **при каждом `updocs`** — минимум:
- `last_updated` (ISO-дата),
- `last_activity` (одна строка: что сделано + ID задачи),
- если закрыта фаза/requirement — обновить `progress.completed_*` и `percent`.

## Шаги

### 1. Определение задачи

- Если после `updocs` указан ID — используй его (нормализация: `5` → `T-0005`, `T-5` → `T-0005`).
- Если ID не указан:
  1. Прочитай `.planning/tasks/INDEX.md`.
  2. Отфильтруй задачи со статусом `done` (включая `.planning/tasks/archive/`).
  3. Покажи последние 5-10 по дате закрытия, жди выбора номером или ID.
- Если задача не в `done` — сообщи: «Задача `T-NNNN` в статусе `<статус>`. `updocs` работает только с `done`. Сначала закрой через `/task review`.» Остановись.
- Прочитай `TASK.md` полностью (frontmatter + все секции).

### 2. Загрузка GSD-документов

Прочитай целиком:
- `.planning/PROJECT.md`
- `.planning/REQUIREMENTS.md`
- `.planning/ROADMAP.md`
- `.planning/STATE.md`

### 3. Анализ задачи и формулировка предложений

На основе `TASK.md`:
- **Цель задачи** — что она на самом деле покрыла.
- **Изменённые файлы** — из `## Журнал выполнения`.
- **Решения** — `## Решения`, `## Доработки` (R1..RN).
- **Связанные задачи** — `related`, `epics`.

Сформируй **предложение по правкам** — четыре блока:

```
PROJECT.md:
  - <предложение или «без изменений»>

REQUIREMENTS.md:
  - <предложение или «без изменений»>

ROADMAP.md:
  - <предложение или «без изменений»>

STATE.md:
  - last_updated → <ISO дата>
  - last_activity → «<строка>»
  - progress → <изменение, если есть>
  - <другие правки>
```

Показывай **конкретные правки** (например «REQUIREMENTS.md: SHIP-02 → `[x]`»), не абстрактные формулировки.

### 4. Интерактивная развилка

По каждому блоку, где предложение неоднозначно, **задай пользователю вопрос**. Типовые случаи:

- **Маппинг task → requirement ID.** «T-0005 покрывает SHIP-02 (click-to-move) полностью, SHIP-03 (плавная ротация) — только rotation damping, без targeting. Отмечаем как `[x]` / оставляем `[ ]` / частично?»
- **Validated vs Active.** «T-0005 доставляет плавное управление — двигать из Active в Validated?»
- **Decimal phase в ROADMAP.** «Цепочка T-0002..T-0005 по смыслу = вставка между Phase 2 и Phase 3. Добавить `Phase 2.4 [INSERTED]: NFE Session Bootstrap & Mouse Input` или не добавлять?»
- **Неизвестный requirement ID.** «Задача реализует X, но я не нашёл соответствующий ID в REQUIREMENTS.md. Пропускаем / добавляем новый ID (предложи формат)?»

Группируй вопросы в **одно сообщение** (нумерованный список), жди ответа одним сообщением. При необходимости follow-up.

Сохраняй ответы пользователя в `TASK.md` → секция `## Обсуждение` с пометкой `(updocs)`:
```
- Q (2026-04-20, updocs): <вопрос>
  A: <ответ>
```

### 5. Применение правок

После уточнений — применяй Edit по каждому документу **точечно**. Для STATE.md допустима правка YAML frontmatter (перезапись нужных ключей).

**Правила редактирования:**
- **Не переформатируй** файл в целом — меняй только нужные строки.
- Дата в `STATE.md:last_updated` — ISO `YYYY-MM-DDT00:00:00.000Z` (UTC-полночь, без точного времени).
- Дата в `STATE.md:last_activity` — `YYYY-MM-DD - <краткое описание> (T-NNNN)`.
- Decimal-фазы в ROADMAP — формат как у существующих: `- [ ] **Phase <N.M>: <Название>** [INSERTED] - <описание>`. Позиция — между целыми N и N+1, сортировка по номеру.
- Если добавляешь новый requirement ID — **подтверди с пользователем номер** (шаг 4).

### 6. Запись в TASK.md

После применения правок — в `## Журнал выполнения` задачи добавь строку:
```
- 2026-04-20 — updocs: обновлены PROJECT/REQUIREMENTS/ROADMAP/STATE.md. Ключевые изменения: <перечень>.
```

(Какие именно документы тронуты — те и упоминай.)

### 7. Финал

Сообщи пользователю:
```
GSD-документы синхронизированы с T-NNNN:
- PROJECT.md: <что изменено или «без изменений»>
- REQUIREMENTS.md: <что изменено>
- ROADMAP.md: <что изменено>
- STATE.md: <что изменено>

Проверь diff и коммить.
```

## Запрещено в этом режиме

- Git-операции (add, commit, push, stash и т.д.).
- Изменять `TASK.md` кроме `## Обсуждение` (ответы Q/A по updocs) и `## Журнал выполнения` (строка про синхронизацию).
- Менять статус задачи (`done` остаётся `done`).
- Переформатировать GSD-документы целиком / рефакторить их структуру без явного запроса.
- Добавлять в GSD-документы ссылки на task-ID (`T-NNNN`) без согласования — task-система параллельна GSD, связи на уровне формулировок, не ID.
- Принимать решения за пользователя по неоднозначным маппингам — всегда спрашивай (шаг 4).
